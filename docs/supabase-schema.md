# Supabase Schema

Supabase Auth is the source of user identity. Application tables reference `auth.users(id)` through a `user_id` column. The MVP migration defaults `user_id` to `auth.uid()` so signed-in clients can usually omit `user_id` on inserts.

## Tables

### `devices`

Tracks devices owned by each authenticated user.

Columns:

- `id`: device ID.
- `user_id`: owner, defaults to `auth.uid()` and references `auth.users(id)`.
- `client_device_key`: stable non-secret key generated by the app for one local install.
- `name`: user-visible device name.
- `platform`: `ios` or `windows`.
- `created_at`: creation timestamp.
- `last_seen_at`: optional last activity timestamp.

Important constraints:

- `(user_id, client_device_key)` is unique so each app install can reuse one device row.
- Device names and client device keys cannot be empty.
- `(id, user_id)` is unique so clipboard rows can reference a source device owned by the same user.

### `clipboard_items`

Stores text clipboard entries owned by each authenticated user.

Columns:

- `id`: clipboard item ID.
- `user_id`: owner, defaults to `auth.uid()` and references `auth.users(id)`.
- `source_device_id`: optional device that created the item.
- `content`: clipboard text.
- `content_type`: defaults to `text/plain`.
- `created_at`: creation timestamp.

Important constraints:

- `content` cannot be empty.
- `content_type` must be `text/plain`.
- `(source_device_id, user_id)` must match a row in `devices` when `source_device_id` is present.
- Deleting a device that has clipboard items is restricted; delete the clipboard items first or keep the device row as history.

## Initial Migration

The SQL migration lives at:

```text
supabase/migrations/0001_initial_schema.sql
```

The migration enables `pgcrypto` with `create extension if not exists pgcrypto;` so `gen_random_uuid()` is available for UUID primary keys.

## Security

Row-level security must be enabled for both tables.

Policy requirements:

- Authenticated users can read, insert, update, and delete their own device rows.
- Authenticated users can read, insert, update, and delete their own clipboard item rows.
- Users cannot access rows owned by other users.
- Anonymous users cannot access clipboard data.

The migration explicitly revokes table access from `anon` and grants table access to `authenticated`. RLS policies still perform the real per-user authorization.

## Verification Checklist

- Create two test users.
- Insert a device and clipboard item as user A.
- Confirm user A can read both rows.
- Confirm user B cannot read user A's rows.
- Confirm anonymous requests cannot read or write clipboard data.
- Confirm a clipboard item cannot reference another user's device as its `source_device_id`.
